# 9-1 新手的解題思維

想像一下：你剛接到一個新的程式專案，螢幕前的你卻不知從何下手。腦中有一堆想法，但就是無法組織成有效的解決方案。這是不是很熟悉的場景？

你並不孤單。每個工程師都曾經歷過這種時刻 —— 一個看似簡單的需求，卻讓你感覺像是站在迷宮入口，不知該往哪個方向邁出第一步。

## 為什麼需要解題思維？

每個程式設計師的職業生涯，都從"不知道怎麼開始"開始。最常見的困境包括：
1. 面對白板時腦中一片空白
2. 寫出的程式碼雜亂無章，難以維護
3. 遇到 bug 時像迷失在迷宮
4. 解決方案難以在其他專案中重複使用

這些困境不是因為你能力不足，而是缺少一個系統化的思考方法。就像建築師在動工前需要完整的藍圖，程式設計師也需要一個清晰的解題框架。這個框架不只是一個工具，更是你在程式設計路上的指南針，幫助你：

* 將模糊的需求轉化為明確的目標
* 把複雜的問題拆解成可管理的小任務
* 建立可重複使用的解決方案模式
* 在遇到瓶頸時找到突破口

讓我們開始學習這個強大的思維工具。

![解題思維框架](images/problem-solving-flow.svg)

## 基本解題步驟

想像你是一位偵探，每個程式問題都是一個待解的謎題。優秀的偵探不會直接跳到結論，而是遵循系統化的調查方法。讓我們一起學習這套方法：

### 1. 定義問題的 I/O（輸入/輸出）

這是解題的第一步，也是最關鍵的一步。就像偵探會先確認案件的基本資訊，我們需要清楚地定義：

- 這個問題要達成什麼目標？（確立終點）
- 現在的狀態是什麼？（確認起點）
- 理想的結果是什麼？（設定期望）
- 有什麼限制條件？（了解邊界）

讓我們看一個實際的例子：

假設問題是「內容創作效率低」：
- 輸入：主題選擇、參考資料、寫作時間
- 輸出：文章數量和品質
- 目標：提高產量同時維持品質
- 限制：時間有限、資源受限

這樣的定義幫助我們將抽象的"效率低"問題，轉化為可以具體衡量和改善的指標。

### 2. 拆解問題

大問題就像一座大山，需要一步步切割成可以搬動的小石頭。這個階段分為三個重要步驟：

1. 數字化當前狀態
   - 目前的寫作速度？（例：8小時/篇）
   - 一篇文章的完成時間？（精確到每個環節）
   - 內容品質評分標準？（設定具體指標）

2. 找出瓶頸
   - 選題耗時（2小時在猶豫主題）
   - 資料搜集慢（資料散亂，整理費時）
   - 內容產出慢（缺乏結構化方法）

3. 定義改善目標
   - 目標產出數量（從每週2篇提升到5篇）
   - 品質要求標準（維持90%以上的讀者滿意度）
   - 時間投入上限（每篇文章最多3小時）

### 3. 規劃解決方案

有了清晰的問題定義和具體的目標，現在我們可以設計解決方案。這個階段需要：

1. 分析現狀流程
2. 識別優化機會
3. 設計改進方案

以內容創作為例：
```
[現狀分析]
選題 (2小時) → 寫作 (4小時) → 修改 (2小時) = 8小時/篇

[問題拆解]
1. 選題慢 → 建立主題庫 + LLM 輔助
   - 預先準備100個主題候選
   - 使用 AI 協助主題評估
   - 建立主題評分機制

2. 寫作慢 → 使用模板 + AI 協作
   - 開發3-5個核心文章模板
   - 設置自動化寫作輔助工具
   - 建立快速成文流程

3. 修改慢 → 自動化檢查 + 智能優化
   - 導入自動校對工具
   - 設置品質檢查清單
   - 使用 AI 輔助潤稿

[優化方案]
選題 (0.5小時) → 寫作 (2小時) → 修改 (0.5小時) = 3小時/篇
```

### 4. 實施計劃

分階段落實優化方案：

1. 第一階段：打基礎（1-2週）
   - 建立學習節奏
     * 週一：挑選本週練習題目
     * 週二：研究解題思路
     * 週三：手寫程式實作
     * 週四：優化程式碼
     * 週五：同學交流討論

2. 第二階段：養成習慣（3-4週）
   - 每日例行
     * 早上：手寫程式練習（1小時）
     * 中午：同學討論解法（30分鐘）
     * 晚上：ORID 反思（30分鐘）

   - 週末整理
     * 週六：整理一週所學（2小時）
       - 畫出概念地圖
       - 歸納解題模式
       - 記錄常見錯誤
     * 週日：預習下週內容（1小時）
       - 閱讀相關文件
       - 準備練習題目
       - 設定學習目標

3. 第三階段：深化學習（5週以後）
   - 週度 ORID
     * O：這週學了哪些新知識點？
     * R：哪些解題過程特別有感觸？
     * I：發現了什麼規律或模式？
     * D：下週要如何調整學習方式？

   - 月度複盤
     * 整理解題心得
     * 分享學習經驗
     * 調整練習難度

### 5. 效果追蹤

設立關鍵指標：

1. 練習成效
   - 每天至少完成 2 個程式題目
   - 每週整理 3 個重要解題模式
   - 每月掌握 1 個核心演算法

2. 理解程度
   - 能用自己的話解釋概念
   - 能畫出概念關係圖
   - 能舉一反三解決類似問題

3. 長期成長
   - 建立解題思維框架
   - 形成個人解題風格
   - 培養持續練習習慣

記住：程式設計的進步來自於大量的練習和深入的思考。重點不是工具和系統，而是實際動手的次數和每次練習後的認真反思。

## 實戰案例：優化學習流程

讓我們用一個所有人都熟悉的場景來實踐解題框架：如何提升程式學習效率？這個案例特別適合新手工程師，因為它直接關係到你的學習曲線。

![學習流程優化案例](images/learning-optimization-flow-visual.svg)

### 1. 定義 I/O

首先，我們需要清晰地定義這個問題的輸入與輸出：

- 輸入：
  - 學習材料（官方文檔、教學視頻、實戰項目）
  - 時間投入（每天可用的學習時間）
  - 學習方法（閱讀、實踐、筆記方式）
  - 學習工具（IDE、筆記軟件、AI 助手）

- 輸出：
  - 知識掌握度（能否解釋核心概念）
  - 實踐能力（能否獨立完成項目）
  - 知識遷移能力（能否舉一反三）
  - 解決問題速度（Debug 和實現功能的效率）

- 目標：
  - 提高學習效率（每週掌握一個新技術要點）
  - 加深理解（能夠向他人清楚解釋）
  - 建立長期記憶（一個月後仍能應用）

- 限制：
  - 時間有限（每天最多 2 小時）
  - 注意力分散（工作學習兼顧）
  - 缺乏即時指導（自學為主）
  - 預算限制（免費資源優先）

### 2. 問題拆解

讓我們深入分析當前的學習痛點：

1. 學習效率低
   - 資料難以理解（專業術語多，概念抽象）
   - 實踐機會少（只讀不寫，難以內化）
   - 缺乏即時反饋（不知道自己哪裡錯）
   - 學習計劃混亂（想到哪學到哪）
   - 基礎概念不牢（新知識難以連結）

2. 知識整合慢
   - 筆記不系統（零散片段，難以回顧）
   - 複習不及時（學完就忘，需要重學）
   - 應用場景少（與實際工作脫節）
   - 知識點分散（沒有形成知識網絡）
   - 缺乏複習機制（被動式學習）

3. 學習成果評估
   - 當前狀態：
     * 完成一個概念學習：2-3天
     * 知識保留率：30%（一週後）
     * 實踐項目完成率：50%
     * Debug 時間：平均 2 小時/問題

   - 目標狀態：
     * 完成一個概念學習：1天
     * 知識保留率：80%（一週後）
     * 實踐項目完成率：90%
     * Debug 時間：平均 30 分鐘/問題

### 3. 解決方案

根據問題拆解，我們設計以下優化方案：


1. 輸入優化（提升學習效率）
   - LLM 輔助理解
     * 使用 ChatGPT 解釋複雜概念
     * AI 生成通俗易懂的類比
     * 多角度解釋技術要點
   
   - 學習資料結構化
     * 建立知識主題清單
     * 設計學習路線圖
     * 準備每日學習清單

   - 自動化筆記流程
     * 使用 Notion 模板
     * 自動化標籤系統
     * 知識點關聯圖譜

2. 處理優化（強化學習過程）
   - 結構化練習系統
     * 每日手寫程式練習（不依賴 AI）
     * 循序漸進的難度安排
     * 重複練習核心概念
   
   - ORID 反思機制
     * Objective：今天學了什麼？
     * Reflective：哪些部分最有感觸？
     * Interpretive：從中學到什麼？
     * Decisional：下次要如何改進？

   - 同儕學習機制
     * 與同學互相 code review
     * 小組討論解題思路
     * 定期分享學習心得

3. 輸出優化（鞏固學習成果）
   - 手寫筆記整理
     * 畫概念關係圖
     * 整理錯誤教訓
     * 記錄解題思路

   - 實戰練習
     * 每週完成指定練習
     * 嘗試不同解題方法
     * 優化程式碼品質

   - 知識分享
     * 教學相長
     * 寫技術筆記
     * 參與技術討論

## 常見誤區

1. 直接跳到解決方案
   - 正確做法：先定義問題的 I/O

2. 想一次解決所有問題
   - 正確做法：找出最大瓶頸優先解決

3. 沒有數字化目標
   - 正確做法：設定可量化的改善指標

4. 忽視自動化機會
   - 正確做法：識別可自動化的環節
